**Independent Review**

The `.d` vs `.raw` asymmetry is basically right, but the wording “zero ripple” for TDF is a little too strong. `TimsId` is indeed an offset table in SQLite, and appending blocks to `analysis.tdf_bin` then recording `tell()` is much cheaper than rewriting an embedded binary index. But TDF still has semantic consistency constraints: frame counts, `NumScans`, MS/MS window tables, calibration/group metadata, and bin block lengths must agree. The asymmetry is not “TDF has no ripple”; it is “TDF localizes physical-address ripple to one row/column while RAW embeds the physical layout in packed file sections.”

The checksum claim is plausible as stated if the crate really recomputes only in `save()` and the checksum window is only the first 10 MiB with the field zeroed. But the doc is slightly overconfident in treating that as a footnote. A valid checksum proves only byte integrity under Thermo’s lightweight check; it does not prove section graph consistency. RawFileReader may accept checksum-valid files and then fail later while materializing trailers, filters, controllers, or chromatograms.

Tier 1 is the right first milestone, but the update list is probably incomplete as written. Repacking scan data and rewriting `ScanIndexEntry.offset` / `data_packet_size` is necessary, and shifting `scantrailer_addr`, `scanparams_addr`, `error_log_addr` plus their 32-bit mirrors is necessary. I would explicitly add these checks/work items:

- Update every RunHeader/data-section boundary field, not only trailing-section starts. If there are stored data-start, data-end, packet-region length, scan-index length, or total packet bytes fields, they must be regenerated.
- Audit all 64-bit and 32-bit mirror addresses. The doc mentions mirrors, but Tier 1 should fail closed unless each mirror is located and updated together.
- Search for an “index of indices” or section directory outside RunHeader. Thermo RAWs often have multiple controller/run structures and late-file directories. If instrument method, tune data, status log, sample info, chromatogram index, trailer index, or error-log pointers are in a global directory, shifting bytes before them may require more than RunHeader edits.
- Recompute per-scan summary fields after payload replacement: peak count, profile/centroid sizes, TIC, base peak m/z/intensity, low/high m/z. The roadmap names ScanIndex summary fields but does not say they are recomputed. RawFileReader/search tools may trust these over packet contents.
- Validate relative vs absolute offsets. `ScanIndexEntry.offset` may be file-absolute, data-section-relative, or controller-relative depending on version. Tier 1 should derive this from existing entries and assert round-trip equality before mutation.
- Treat padding/alignment as a first-class invariant. If packets or sections are aligned, naive concatenation can shift later addresses by legal-but-unexpected values.
- Audit trailer `GenericRecord` internals. If a GenericRecord contains only local lengths and typed values, moving the stream is fine. If it contains string/table offsets, calibration references, or intra-stream pointers, Tier 1 must either preserve byte-identical records or adjust internal offsets.
- Check status/log/instrument-method/tune-data addresses even if Tier 1 does not restructure them. If they live after the data section, they shift. If they live before it, they do not. The plan currently names only `scantrailer`, `scanparams`, and `error_log`.

Tier 1 should therefore be framed as “rebuild the complete section-address graph after the data region,” not just “shift three trailing addresses.”

Tier 2 is materially riskier than the roadmap’s tone suggests. Full scan-event records are not just filter text ingredients. RawFileReader may validate or derive behavior from analyzer type, scan mode, dependent scan relationships, precursor metadata, activation, polarity, detector class, resolution, AGC/injection time-like fields, centroid/profile flags, mass ranges, method segment IDs, trailer-extra keys, and controller/run consistency. Unknown stride fields are a trap because skipped fields in your reader may still be required by Thermo’s reader, ProteoWizard, vendor DLL chromatogram extraction, or downstream search engines. Copying a same-MS-level skeleton and patching known fields is the correct mitigation, but then “arbitrary layout” is only safe for layouts composable from known real event archetypes.

Sequencing is right: Tier 1 first. It attacks the real fidelity loss, avoids inventing new scan events, and creates the section-rebuild machinery Tier 2 needs anyway.

There may be a cheaper partial path to arbitrary peaks: preallocate oversized packets in a template, or rewrite within fixed-size padded packets. That avoids shifting trailing sections and RunHeader pointers. It is attractive for production simulation if you can tolerate larger files and define a hard upper bound, e.g. reserve N centroid slots per MS1/MS2 and set actual count smaller. But it only works if packet readers honor the internal peak count and ignore unused bytes, and if `data_packet_size` can remain the padded size without confusing parsers. I would prototype this before full Tier 1: create one scan with padded packet size, actual count smaller, checksum valid, then test RawFileReader, ProteoWizard, Sage, and DIANN. If accepted, padding is a cheap bridge; it is not a substitute for arbitrary layouts.

Main overconfidence: “bounded, mechanical” and “same kind of work as Bruker compression.” RAW section graphs may have duplicate directories and version-specific mirrors. Main underconfidence: Tier 1 is still very achievable if you make it evidence-driven: parse all known absolute offsets before and after, classify every pointer into moved/unmoved regions, regenerate summaries, and test with official RawFileReader plus downstream tools.
